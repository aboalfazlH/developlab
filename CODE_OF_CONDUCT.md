# Code of Conduct

---

## English Version

### 1. Purpose

This document defines a set of professional behavior standards to ensure a respectful, secure, and productive environment for all developers, contributors, and users involved in this project.

---

### 2. Expected Behavior

#### 2.1 Respectful Communication

- Interact with professionalism and clarity.
- Avoid personal attacks, sarcasm, or disrespectful language.
- Keep discussions technical, not personal.

#### 2.2 Collaboration & Support

- Provide constructive and helpful feedback during code reviews.
- Share knowledge transparently; avoid intentional withholding of information.
- Maintain clear and documented communication.

#### 2.3 Responsibility & Accountability

- Each developer is responsible for the quality, security, and maintainability of their code.
- Avoid pushing changes directly to protected branches (e.g., `main`).
- Breaking changes must be documented and communicated in advance.

#### 2.4 Technical Standards

- Follow clean code principles and project coding guidelines.
- Keep commits small, readable, and meaningful.
- Include required tests and validations before submitting PRs.

---

### 3. Unacceptable Behavior

#### 3.1 Harassment & Abuse

- Insults, threats, discrimination, or any form of harassment.
- Offensive comments related to race, gender, religion, political views, etc.

#### 3.2 Irresponsible Technical Practices

- Pushing secrets such as passwords or API keys into the repository.
- Introducing security risks intentionally or through neglect.
- Making undocumented breaking changes.

#### 3.3 Misuse of Communication Channels

- Spamming issues or PRs with irrelevant content.
- Avoiding necessary responses or blocking workflow intentionally.

---

### 4. Reporting Violations

To report violations:

- Submit a private report through the official project communication channel.
- Include:
  - Description of the behavior
  - Time, context, involved contributors (if applicable)
  - Evidence (logs, messages, screenshots)

Reporters will be protected from retaliation.

---

### 5. Consequences of Violations

- **Minor violation:** Verbal/written warning.
- **Moderate violation:** Restricted repository access, PR blocks, formal meeting.
- **Severe violation:** Removal from contributors, access revocation, or project ban.

All actions will be documented and reviewed by the project maintainers.

---

### 6. Repository & Code Standards

#### 6.1 Commit Messages

- Use clear, concise, and meaningful commit messages.
- Prefer present tense ("Add feature", not "Added feature").
- Reference related issues or tasks when applicable.
- Example: `Fix login bug affecting session timeout (#45)`

#### 6.2 Pull Requests (PR)

- Keep PRs small, focused, and self-contained.
- Include a clear description of the changes and reasoning.
- Ensure all tests pass before requesting a review.
- Tag reviewers and related issues for context.

#### 6.3 Branching Strategy

- Follow the branching model defined by the project (e.g., `main`, `develop`, `feature/*`, `bugfix/*`).
- Avoid pushing directly to protected branches (`main`, `develop`).
- Merge only via PR after proper code review.

#### 6.4 Code Reviews

- Review code constructively, focusing on logic, readability, and maintainability.
- Suggest improvements politely and provide examples when possible.
- Approve PRs only when code meets project standards and tests pass.

#### 6.5 Testing & Validation

- Include unit, integration, or functional tests as appropriate.
- Run all tests locally before creating a PR.
- Document any known limitations or issues in the PR description.

## نسخه فارسی

### 1. هدف

این سند مجموعه‌ای از استانداردهای رفتار حرفه‌ای را تعریف می‌کند تا محیطی محترمانه، امن و تولیدمحور برای تمامی توسعه‌دهندگان، مشارکت‌کنندگان و کاربران پروژه فراهم شود.

---

### 2. رفتار مورد انتظار

#### 2.1 ارتباط محترمانه

- با حرفه‌ای‌گری و شفافیت تعامل کنید.
- از حملات شخصی، کنایه یا زبان بی‌احترامی پرهیز کنید.
- مباحث فنی را فنی نگه دارید و شخصی نشود.

#### 2.2 همکاری و پشتیبانی

- بازخورد سازنده و مفید در بازبینی کد ارائه دهید.
- دانش خود را شفاف به اشتراک بگذارید و اطلاعات را عمداً پنهان نکنید.
- ارتباطات را شفاف و مستند نگه دارید.

#### 2.3 مسئولیت و پاسخگویی

- هر توسعه‌دهنده مسئول کیفیت، امنیت و نگهداری کد خود است.
- از ارسال مستقیم تغییرات به برنچ‌های محافظت‌شده (مثل `main`) خودداری کنید.
- تغییرات بحرانی باید مستند و از قبل اطلاع‌رسانی شوند.

#### 2.4 استانداردهای فنی

- از اصول کد تمیز و دستورالعمل‌های کدنویسی پروژه پیروی کنید.
- کامیت‌ها را کوچک، خوانا و معنا‌دار نگه دارید.
- قبل از ارسال PR، تست‌ها و اعتبارسنجی‌های لازم را انجام دهید.

---

### 3. رفتار غیرقابل قبول

#### 3.1 آزار و اذیت

- توهین، تهدید، تبعیض یا هر نوع آزار و اذیت.
- نظرات توهین‌آمیز مرتبط با نژاد، جنسیت، مذهب، دیدگاه‌های سیاسی و غیره.

#### 3.2 شیوه‌های فنی غیرمسئولانه

- ارسال رمز عبور یا API Key به مخزن.
- ایجاد خطرات امنیتی عمداً یا از روی غفلت.
- اعمال تغییرات بحرانی بدون مستندسازی.

#### 3.3 سوءاستفاده از کانال‌های ارتباطی

- ارسال محتوای نامرتبط در Issue یا PR.
- عدم پاسخ‌دهی لازم یا جلوگیری عمدی از جریان کاری.

---

### 4. گزارش تخلفات

برای گزارش تخلفات:

- گزارش خصوصی از طریق کانال رسمی ارتباطی پروژه ارسال کنید.
- شامل موارد زیر باشد:
  - شرح رفتار
  - زمان، زمینه، مشارکت‌کنندگان در صورت وجود
  - شواهد (لاگ‌ها، پیام‌ها، اسکرین‌شات‌ها)

گزارش‌دهندگان از هرگونه تلافی محافظت خواهند شد.

---

### 5. پیامدهای تخلفات

- **تخلف جزئی:** هشدار کتبی یا شفاهی.
- **تخلف متوسط:** محدودیت دسترسی به مخزن، بلاک شدن PR، جلسه رسمی.
- **تخلف شدید:** حذف از مشارکت‌کنندگان، لغو دسترسی یا محرومیت از پروژه.

تمام اقدامات مستند و توسط مدیران پروژه بررسی می‌شود.

---

### 6. استانداردهای ریپازیتوری و کد

#### 6.1 پیام‌های کامیت

- پیام‌های کامیت باید واضح، کوتاه و معنا‌دار باشند.
- از زمان حال استفاده کنید ("افزودن قابلیت" به جای "قابلیت افزوده شد").
- در صورت نیاز، به مسائل یا تسک‌های مرتبط اشاره کنید.
- مثال: `رفع باگ ورود که باعث تایم‌اوت جلسه می‌شد (#45)`

#### 6.2 درخواست‌های کشش (Pull Request)

- PRها را کوچک، متمرکز و مستقل نگه دارید.
- توضیح واضحی از تغییرات و دلایل آن ارائه دهید.
- قبل از درخواست بررسی، اطمینان حاصل کنید که تمام تست‌ها عبور می‌کنند.
- بازبین‌ها و مسائل مرتبط را برای ارائه زمینه تگ کنید.

#### 6.3 استراتژی برنچ‌ها

- از مدل برنچ پروژه پیروی کنید (مثلاً `main`، `develop`، `feature/*`، `bugfix/*`).
- از ارسال مستقیم به برنچ‌های محافظت‌شده (`main`، `develop`) خودداری کنید.
- تنها از طریق PR و پس از بررسی کد مناسب، merge انجام شود.

#### 6.4 بازبینی کد

- بازبینی کد را به شکل سازنده انجام دهید و روی منطق، خوانایی و نگهداری تمرکز کنید.
- پیشنهادهای بهبود را مودبانه ارائه دهید و در صورت امکان مثال بیاورید.
- PRها را تنها زمانی تأیید کنید که کد مطابق استانداردهای پروژه باشد و تست‌ها موفقیت‌آمیز باشند.

#### 6.5 تست و اعتبارسنجی

- شامل تست‌های واحد، یکپارچه یا عملکردی باشد، در صورت نیاز.
- قبل از ایجاد PR، تمام تست‌ها را به صورت محلی اجرا کنید.
- هر محدودیت یا مشکل شناخته‌شده را در توضیحات PR مستند کنید.
